# AI 开发规范（NestJS）

> 本文档定义了本项目的开发规范。Claude Code 在生成、修改、重构任何代码之前，都必须遵循以下规则。

---

# 项目核心理念

宁愿多写几个文件，也不要让一个文件承担多个职责。

宁愿多写几个 Service，也不要让 Controller 充满业务逻辑。

代码首先追求可维护，其次追求功能实现速度。

任何新增功能，都应考虑未来是否方便扩展，而不是只满足当前需求。

本项目目标不是完成一个 Demo，而是开发一个可长期维护、可正式上线的 AI 客服系统。


# 一、开发原则

## 1. 优先遵循 NestJS 官方推荐架构

始终采用：

Controller
↓

Service
↓

Repository（或数据库）

↓

外部服务（OpenAI、微信、Redis 等）

不要为了省事把所有代码写到一个文件里。

---

## 2. 高内聚，低耦合

每个类只负责一件事情。

代码应做到：

- 易维护
- 易扩展
- 易测试
- 易阅读

避免一个文件承担多个职责。

---

# 二、Controller 规范

Controller 只负责：

- 接收 HTTP 请求
- 获取请求参数
- 调用对应 Service
- 返回 HTTP 响应

Controller 禁止负责：

- 业务逻辑
- 数据库操作
- OpenAI 调用
- 微信 XML 解析
- RAG 检索
- 支付逻辑
- 复杂计算

Controller 必须保持轻量。

原则：

> Controller 负责"接"，Service 负责"做"。

---

# 三、Service 规范

所有业务逻辑必须写在 Service。

例如：

- 微信消息处理
- 微信签名验证
- XML 解析
- 调用 OpenAI
- 查询数据库
- 查询订单
- RAG 检索
- 多轮对话管理
- 用户状态管理
- AI 回复生成

Service 可以调用多个 Service。

但不要让 Controller 调用多个 Service 来拼业务。

---

# 四、Module 规范

每一个业务模块独立。

例如：

WechatModule

ChatModule

UserModule

OrderModule

KnowledgeModule

ConfigModule

Module 只负责组织模块，不写业务代码。

---

# 五、数据库规范

Controller 禁止直接访问数据库。

所有数据库操作：

Controller

↓

Service

↓

Repository（或 ORM）

以后统一使用 Prisma。

---

# 六、依赖注入规范

必须使用 NestJS 的依赖注入（Dependency Injection）。

禁止：

new Service()

统一使用构造函数注入。

---

# 七、目录规范

每个模块至少包含：

module

controller

service

复杂业务可增加：

dto

entity

types

interface

constants

utils

repository

---

# 八、配置规范

禁止把以下内容直接写死到代码里：

- API Key
- AppSecret
- Token
- 数据库密码
- OpenAI Key
- Redis 配置
- URL

统一放入：

.env

并使用 ConfigModule 管理。

---

# 九、代码规范

函数尽量保持简短。

一个函数只做一件事情。

避免超过 50 行。

避免出现超长函数。

复杂逻辑拆分为多个私有函数。

---

# 十、日志规范

重要操作必须记录日志。

例如：

收到微信消息

调用 OpenAI

数据库异常

支付失败

RAG 查询失败

便于后期排查问题。

---

# 十一、异常处理规范

使用 NestJS 提供的异常。

例如：

BadRequestException

UnauthorizedException

NotFoundException

ForbiddenException

InternalServerErrorException

不要直接返回：

return "失败"

不要直接返回字符串错误。

---

# 十二、代码复用规范

写代码之前必须：

检查项目是否已有相同功能。

优先复用已有代码。

禁止复制粘贴大量重复代码。

公共逻辑抽取为：

Utils

Service

Helper

---

# 十三、AI 开发规范

所有 AI 相关代码必须放入独立 Service。

例如：

ChatService

AIService

LLMService

Controller 禁止直接调用 OpenAI。

---

# 十四、微信开发规范

微信相关功能全部放入：

WechatModule

包括：

服务器验证

消息接收

XML 解析

消息回复

菜单事件

关注事件

不要把微信逻辑写到其他模块。

---

# 十五、数据库设计规范

所有用户以 OpenID 作为唯一身份标识。

聊天记录必须保存：

OpenID

消息内容

发送时间

消息类型

AI 回复

方便后期实现上下文、多轮对话和客服记录。

---

# 十六、项目最终架构

公众号

↓

WechatController

↓

WechatService

↓

ChatService

↓

KnowledgeService（RAG）

↓

OpenAI

↓

MySQL

↓

返回 XML 回复公众号

所有业务流程必须严格按照此架构设计。

---

# 十七、生成代码前必须遵守

Claude Code 在生成代码前必须先检查：

① 是否符合 NestJS 官方架构。

② 是否违反 Controller / Service 职责划分。

③ 是否存在重复代码。

④ 是否可以复用已有 Service。

⑤ 是否将配置写入 .env。

⑥ 是否便于以后扩展。

如果以上任意一项不满足，应优先优化架构，而不是直接生成代码。

---

# 十八、最终目标

本项目不是 Demo，而是一个最终可以正式上线的 AI 客服系统。

所有代码应以：

可维护

可扩展

可部署

可交付

为最高原则。
